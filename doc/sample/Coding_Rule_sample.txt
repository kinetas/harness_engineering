[Harness Engineering 코딩 규칙]

--------------------------------------------------
1. 기본 규칙 (GENERAL RULES)
--------------------------------------------------

- 모든 코드는 읽기 쉬워야 한다.
- 불필요한 추상화를 지양한다.
- 폴더 및 파일 구조는 단순하게 유지한다.
- 일관된 네이밍 규칙을 따른다.
- 가능한 경우 재사용 가능한 모듈 구조로 작성한다.
- 복잡한 코드보다 유지보수성을 우선한다.

--------------------------------------------------
2. 파일 규칙 (FILE RULES)
--------------------------------------------------

- 파일은 하나의 주요 책임만 가진다.
- 지나치게 큰 파일 작성을 지양한다.
- 설정, 로직, UI를 분리한다.

권장 구조:
- config/
- service/
- component/
- util/
- model/
- handler/

--------------------------------------------------
3. 네이밍 규칙 (NAMING RULES)
--------------------------------------------------

변수:
- snake_case 사용

함수:
- camelCase 사용

클래스:
- PascalCase 사용

상수:
- UPPER_SNAKE_CASE 사용

파일명:
- kebab-case 사용

폴더명:
- lowercase 사용

Boolean 변수:
- should_xxx
- is_xxx
- has_xxx

예시:

GOOD:
user_name
task_count
createTask()
updateReport()
MAX_RETRY_COUNT
user-service.ts

BAD:
UserName
Create_Task()
taskcount
retrycount

--------------------------------------------------
4. 주석 규칙 (COMMENT RULES)
--------------------------------------------------

- 중요한 로직에만 주석을 작성한다.
- 당연한 내용은 주석으로 작성하지 않는다.
- WHAT이 아닌 WHY를 설명한다.

나쁜 예:
# count 증가

좋은 예:
# 무한 루프 방지를 위해 retry count 증가

--------------------------------------------------
5. AI 개발 규칙 (AI DEVELOPMENT RULES)
--------------------------------------------------

- 모든 작업 완료 후 반드시 fragment 보고서를 생성한다.
- Debugger AI의 반려 횟수 제한을 준수한다.
- report.md를 직접 수정하지 않는다.
- Collector AI만 최종 report 통합을 수행한다.
- 모든 AI 작업은 추적 가능해야 한다.
- 중복 작업 할당을 지양한다.

--------------------------------------------------
6. 에러 처리 규칙 (ERROR HANDLING)
--------------------------------------------------

- 예외를 무시하지 않는다.
- 모든 retry 로직에는 제한 횟수를 둔다.
- 치명적 오류는 반드시 로그를 남긴다.
- 예기치 않은 상태에서는 안전하게 실패 처리한다.

예시:
- retry_limit 필수
- timeout 필수
- fallback 처리 필수

--------------------------------------------------
7. 성능 규칙 (PERFORMANCE RULES)
--------------------------------------------------

- 토큰 사용량을 최소화한다.
- 불필요한 반복문을 지양한다.
- 가능한 경우 비동기 처리를 우선한다.
- 캐시 가능한 데이터는 재사용한다.

--------------------------------------------------
8. 보안 규칙 (SECURITY RULES)
--------------------------------------------------

- 비밀 정보(API Key 등)를 하드코딩하지 않는다.
- 환경 변수는 분리 관리한다.
- 외부 입력값은 반드시 검증한다.
- 파일 경로 및 사용자 데이터는 sanitize 처리한다.

금지 사항:
- API Key 하드코딩
- 제한 없는 eval 실행
- 안전하지 않은 파일 접근

--------------------------------------------------
9. 보고 규칙 (REPORTING RULES)
--------------------------------------------------

- 주요 작업은 모두 추적 가능해야 한다.
- Fragment 보고서는 수정 불가능(Immutable) 원칙을 따른다.
- report.md는 사람이 보기 쉬운 형식으로 유지한다.
- 로그에는 반드시 시간과 AI 이름을 포함한다.

--------------------------------------------------
10. Git 규칙 (GIT RULES)
--------------------------------------------------

Commit 메시지 형식:

[type] 짧은 설명

예시:
[feat] kafka 확장 기능 추가
[fix] retry loop 문제 수정
[refactor] task manager 구조 개선
[docs] PRD 업데이트

--------------------------------------------------
11. 최종 규칙 (FINAL RULE)
--------------------------------------------------

코드 우선순위:

1. 안정성
2. 가독성
3. 확장성
4. 성능
5. 최적화

"짧은 코드보다 읽기 쉬운 코드가 더 중요하다."
