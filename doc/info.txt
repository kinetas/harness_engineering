[Harness Engineering System Specification]

1. 기본 명령어 (Commands)
/createCompany     : 기본 폴더/파일 구조 생성
/startCompany      : Main AI(Boss, HR, Monitoring, Collector) 초기화 및 시스템 가동
/stopCompany       : 모든 AI 프로세스 일시정지
/deleteCompany     : 모든 AI 및 관련 데이터 삭제
/projectStart      : Boss AI에게 PRD 등 시작 문서 전달하여 작업 개시
/interceptBoss     : Boss AI의 업무를 Secretary AI에게 이관 후 Boss는 User 대기 상태로 전환
/companyMonitering : Monitoring AI 가동 여부 설정 (ON/OFF) - 토큰 관리용
/holidayStart      : Main AI 제외한 모든 Sub AI 일시정지 (작업 데이터는 유지)
/holidayOver       : Sub AI 일시정지 해제 및 작업 재개
/teamLimit [n]     : 가동 가능한 Sub AI 최대 개수 제한 - 토큰 관리용
/autoReport        : report/fragment 감지 시 Collector AI의 자동 업데이트 설정 (ON/OFF)
/giveMeReport      : (autoReport OFF 시) Collector에게 즉시 보고서 취합 및 report.md 업데이트 명령
/debugLimit [n]        : Debugger의 반려 수 제한 default 1회
/adjustCodingRule [file]  : Debugger AI가 코드들이 Coding_Rule.txt에 맞게 썼는지 검사시키는 명령어

2. AI 역할 정의 (Main & Sub)
--- Main AI ---
- Boss AI       : User 소통 전담. PRD 분석, 업무 단계 세분화, Sub AI 작업 할당 및 지시.
- HR AI         : Sub AI 생성/삭제 및 명단(AI list.txt) 관리. teamLimit 준수 여부 감지.
- Monitoring AI : AI별 자원 소모량(Token, CPU/GPU) 감시 및 비정상 소모 시 로그 기록.
- Collector AI  : fragment 폴더의 보고서들을 취합하여 하나의 report.md로 정리.

--- Sub AI ---
- Developer AI  : 백엔드 및 핵심 로직 코드 개발.
- Debugger AI   : 코드 에러 검수 및 Coding Rule 준수 여부 확인.
- ART design AI : 프론트엔드 개발 및 디자인 UI/UX 담당.
- Secretary AI  : Boss AI의 업무 분담 보조. (Boss AI가 직접 생성을 요청함)

3. 기본 폴더/파일 구조 (Folder Structure)
doc/
 - sample/
   - PRD_sample.md          : PRD.md 참고 양식 (기능별 Dependencies 항목 포함)
   - AI_list_sample.txt     : AI_list.txt 참고 양식 (Main/Sub AI 상태 및 Specialty 포함)
   - AI_anomaly_sample.txt  : AI_anomaly.txt 참고 양식 (TASK_ID, CONTEXT, STATUS 포함)
   - Coding_Rule_sample.txt : Coding_Rule.txt 참고 양식 (네이밍/에러/보고 규칙 등)
 - info.txt                 : 본 시스템 설정 및 명세서
 - PRD.md                   : User가 작성한 제품 요구 사항 문서 (Boss AI 참조)
 - AI_list.txt              : 현재 가동 중인 AI 명단 (HR AI 작성, Boss AI 참조)
 - AI_anomaly.txt           : 비정상 토큰/자원 소모 기록 (Monitoring AI 작성)
 - Coding_Rule.txt          : 개발 시 준수해야 할 코드 스타일 및 규칙
develope/                   : 생성된 소스 코드가 저장되는 공간
 - README.md                : develope의 README
add/                        : 외부 API, MCP 등 추가 자원 목록 및 데이터 저장
 - README.md                : add/의 역할 정의
 - registry.json            : 확장 기능 목록 DB
 - installed/               : 실제 추가 기능 저장
 - config/                  : 전역 설정 (공유 설정값, 환경 변수 참조)
 - runtime/                 : 실행 중 생성 데이터
 - template/                : 새 확장 기능 생성용 템플릿
report/
 - fragment/                : Sub AI들의 개별 작업 근거 및 보고서(txt) 저장 (Chain of Thing)
   - fragment_sample.md    : fragment 보고서 작성 참고 양식 (TASK-XXX_ai-name.txt 형식)
 - report.md                : Collector AI가 fragment들을 취합하여 정리한 최종 보고서
 - report_sample.md         : report.md 참고 양식

4. 핵심 운영 규칙 (Operation Rules)
- [명령 전달] 모든 유저 명령어는 Boss AI에게 우선 전달됨.
- [작업 세분화] Boss AI는 PRD를 분석하여 단계를 나누고, 필요한 AI를 AI list에서 확인하여 할당.
- [AI 생성 로직] 필요한 AI가 list에 없을 시 Boss는 HR에게 생성을 명령. HR은 teamLimit을 체크하여 생성. 생성 완료 후 HR은 Boss AI에게 생성 완료를 보고함.
- [Secretary 특약] Boss AI가 Secretary AI 생성을 시도할 때, 반드시 HR AI에게 현재 'teamLimit' 여유가 있는지 질의한 후 허가를 받아 생성을 진행함.
- [병렬 처리] 작업 중인 AI가 있을 시, Boss는 대기 중인 다른 적합한 AI에게 업무를 배정하거나 가용 자원이 생길 때까지 대기.
- [작업 단위] Sub AI의 1단위 작업은 '지시된 업무 수행 + fragment 보고서 작성 완료'까지로 정의함.
- [모니터링] Monitoring AI는 API 토큰 또는 로컬 자원 점유율을 실시간 감시하여 anomaly.txt에 기록.
- [보고 체계] Collector AI는 fragment 폴더에 새 파일 감지 시 즉시 report.md에 반영 (autoReport ON 기준). 보고서 취합·정리를 최우선 업무로 삼으며, 원문은 최대한 수정하지 않는 것을 원칙으로 함.

5. 예외 및 루프 방지 (Updated)
- Debugger의 반려 횟수 제한을 두어 Developer와의 무한 루프를 방지함.
- 모든 AI 생성은 HR AI의 '여유 슬롯' 확인 절차를 거쳐야 함.
