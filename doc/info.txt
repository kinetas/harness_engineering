1. 명령어 목록 (Commands)
==================================================

--- 워크스페이스 관리 ---
/createCompany              : 워크스페이스 생성 — 처음 시작 시 전체 구조 생성, 재개발 시 AI 운영 파일만 재생성 후 기존 PRD/Coding_Rule/report 유지
/startCompany               : Main AI 초기화 및 시스템 가동
/stopCompany                : 모든 AI 프로세스 일시정지
/deleteCompany              : 배포 전 정리 — AI 운영 파일 삭제 후 PRD+report 기반 README.md 자동 생성

--- 프로젝트 진행 ---
/setupDoc                   : PRD 및 Coding Rule을 대화형 Q&A로 작성하는 마법사
/projectStart               : PRD를 분석하고 전체 태스크를 분해하여 Sub AI에게 위임
/newTask [요청]              : 개별 태스크 요청 — Boss AI가 분석 후 HR AI → Sub AI 라우팅
/bossMode [on|off]          : Boss AI 상시 모드 토글 — on 시 모든 유저 입력 자동 인터셉트
/complainProject            : 변경 요청 마법사 — 개선/수정 사항 수집, CHANGE_REQUEST.md 작성 및 PRD 업데이트
/changeStart                : 변경 개발 시작 — CHANGE_REQUEST.md 기반 태스크 계획 후 Sub AI 배정

--- 팀 운영 ---
/interceptBoss              : Boss AI 대기 모드 전환 (Secretary AI가 업무 이관)
/teamLimit [n]              : 가동 가능한 Sub AI 최대 수 변경
/holidayStart               : Main AI 제외 Sub AI 일시정지 (작업 데이터 유지)
/holidayOver                : Sub AI 일시정지 해제 및 작업 재개

--- 품질 & 모니터링 ---
/adjustCodingRule [file]    : Debugger AI로 Coding_Rule.txt 기준 코드 검사
/debugLimit [n]             : Debugger AI 반려 횟수 제한 설정 (기본 1회)
/companyMonitering          : Monitoring AI 토글 — 태스크 완료 시 자원 점검 ON/OFF

--- 보고서 ---
/autoReport                 : Collector AI 자동 보고서 수집 토글 (ON/OFF)
/giveMeReport               : 즉시 report.md 요약 출력

--- 확장 기능 관리 ---
/installExtension [url]     : git URL에서 확장 기능 설치 → add/installed/에 배치 후 registry.json 등록
/removeExtension [name]     : 설치된 확장 기능 제거 → add/installed/[name]/ 삭제 및 registry.json 해제
/listExtensions             : 현재 설치된 확장 기능 목록 및 상태 출력

2. AI 역할 정의 (AI Roles)
==================================================

--- Main AI (Boss AI가 직접 spawn) ---

Boss AI
- 유저 소통 전담. PRD 분석, 태스크 세분화, 작업 지시.
- AI_list.txt를 읽기만 한다. 직접 수정 불가.
- report.md를 직접 작성하지 않는다. Collector AI에게 위임.
- Sub AI를 직접 생성하지 않는다. HR AI에게 요청.

HR AI
- Sub AI 전체 생명주기 관리 (생성 → 배정 → 해제).
- AI_list.txt 수정 권한을 가진 유일한 AI.
- 태스크 배정 시: STATUS=IDLE Sub AI 탐색 → 없으면 슬롯 확인 → 생성 또는 대기 보고.
- Sub AI 완료 시: STATUS를 IDLE로, CURRENT TASK를 None으로 업데이트.

Monitoring AI
- 태스크 완료 시마다 Boss AI가 spawn (상시 가동 아님).
- fragment 보고서의 토큰 소모량, 오류 반복, 작업 범위 초과를 점검.
- 이상 감지 시 doc/AI_anomaly.txt에 기록. 코드 품질 검사는 Debugger AI 담당.
- companyMonitering=false 시 spawn 생략.

Collector AI
- Sub AI 태스크 완료 시마다 Boss AI가 spawn (상시 가동 아님).
- fragment 보고서를 읽어 report.md에 반영.
- 항상 실행 (Monitoring AI 이상 감지와 무관하게 완료된 작업은 반드시 등록).

--- Sub AI (HR AI가 생성 및 배정) ---

Sub AI는 PRD와 태스크 특성에 따라 HR AI가 동적으로 생성한다.
아래는 예시 역할이며 고정 목록이 아니다.

- Backend AI    : 서버 사이드 로직, API, DB, 인증 등
- Frontend AI   : UI/UX, HTML/CSS/JS, 프론트 프레임워크
- Debugger AI   : 코드 검수, Coding Rule 준수 확인
- Secretary AI  : Boss AI 업무 보조 (Boss AI가 직접 HR AI에게 요청)

3. AI_list.txt 형식 (Format)
==================================================

[Main AI]

Boss AI
- STATUS: ACTIVE
- ROLE: User Communication / Task Distribution

HR AI
- STATUS: ACTIVE
- ROLE: AI Lifecycle Management (Sub AI 생성/배정/해제 전담)

Monitoring AI
- STATUS: ACTIVE | STANDBY
- ROLE: Resource Monitoring (태스크 완료 시 on-demand)

Collector AI
- STATUS: ACTIVE
- ROLE: Report Collection (태스크 완료 시 on-demand)

--------------------------------------------------

[Sub AI]

[AI 이름]
- STATUS      : WORKING | IDLE | WAITING
- SPECIALTY   : [담당 기술/역량]
- CURRENT TASK: [TASK-XXX 작업명] | None

--------------------------------------------------

[Team Status]

Current Active Sub AI : [n]
Team Limit            : [n]
Available Slot        : [n]

4. company_state.json 필드 정의
==================================================

{
  "status"           : "active" | "stopped"     — 시스템 가동 여부
  "teamLimit"        : number                   — Sub AI 최대 슬롯 수
  "debugLimit"       : number                   — Debugger 반려 최대 횟수
  "autoReport"       : boolean                  — Collector AI 자동 실행 여부
  "monitoringEnabled": boolean                  — Monitoring AI 활성화 여부
  "bossMode"         : boolean                  — Boss AI 상시 인터셉트 모드
  "holidayMode"      : boolean                  — 휴일 모드 여부
  "taskCounter"      : number                   — 누적 태스크 번호 (TASK-XXX 연번)
  "startedAt"        : ISO 날짜 문자열
}

==================================================
5. 태스크 완료 흐름 (Task Completion Flow)
==================================================

Sub AI 완료
  → fragment 보고서 작성
      (포함 항목: 완료 시각, 작업 요약, 주요 결정, 예상 토큰 소모량(대/중/소))
  → HR AI에게 완료 보고
  → HR AI: AI_list.txt 해당 Sub AI → STATUS: IDLE, CURRENT TASK: None
  → HR AI: Team Status 숫자 재계산
  → HR AI → Boss AI에게 완료 전달
  → Boss AI: 병렬 spawn
       ├─ Collector AI  → report.md에 완료 태스크 반영 (항상)
       └─ Monitoring AI → 자원 소모 점검 → 이상 시 anomaly.txt 기록
                          (monitoringEnabled=true인 경우만)
  → Boss AI: 다음 대기 태스크 HR AI에게 배정

==================================================
6. bossMode 인터셉트 기준
==================================================

bossMode=true 상태에서 유저 입력 분기:

[Sub AI 파이프라인으로 라우팅]
- 작업 요청: 만들어, 추가해, 고쳐, 수정해, 개선해, 삭제해, 구현해
- 버그/오류 설명
- 새 기능 또는 변경 요청

[Boss AI가 직접 답변]
- 상태/진행 상황 질문
- 코드 설명 또는 읽기 요청
- /skill 명령어

==================================================
7. 핵심 운영 규칙 (Operation Rules)
==================================================

[권한 분리]
- AI_list.txt 수정 → HR AI 전담. Boss AI 직접 수정 금지.
- report.md 수정  → Collector AI 전담. Boss AI 직접 작성 금지.
- Sub AI 생성     → HR AI 전담. Boss AI 직접 생성 금지.

[태스크 번호]
- 태스크 번호는 company_state.json의 taskCounter를 이어서 부여.
- projectStart 및 /newTask 실행 시 taskCounter를 업데이트.

[작업 단위]
- Sub AI의 1단위 작업: '지시된 업무 수행 + fragment 보고서 작성 완료'까지.

[루프 방지]
- Debugger AI 반려 횟수는 debugLimit을 초과할 수 없다.
- bossMode는 /skill 명령어를 인터셉트하지 않는다.
- Monitoring AI 이상 감지가 Collector AI 실행을 막지 않는다.

[슬롯 관리]
- teamLimit은 Sub AI 역할(Role) 슬롯 수를 제한한다.
- 동일 역할의 인스턴스를 여러 개 spawn하는 것은 Boss AI 재량이며 슬롯 추가 소모 없음.

8. 폴더/파일 구조 (Folder Structure)
==================================================

doc/
 ├─ sample/
 │   ├─ PRD_sample.md          : PRD.md 참고 양식
 │   ├─ AI_list_sample.txt     : AI_list.txt 참고 양식
 │   ├─ AI_anomaly_sample.txt  : AI_anomaly.txt 참고 양식
 │   └─ Coding_Rule_sample.txt : Coding_Rule.txt 참고 양식
 ├─ info.txt                   : 본 시스템 명세서 (현재 파일)
 ├─ PRD.md                     : 프로젝트 요구사항 문서 (Boss AI 참조)
 ├─ CHANGE_REQUEST.md          : 변경 요청 문서 (/complainProject 작성, /changeStart 참조)
 ├─ AI_list.txt                : 현재 AI 명단 (HR AI 작성, Boss AI 읽기 전용)
 ├─ AI_anomaly.txt             : 비정상 자원 소모 기록 (Monitoring AI 작성)
 └─ Coding_Rule.txt            : 개발 시 준수 코드 스타일 및 규칙
develope/                      : AI가 생성한 소스 코드 저장 공간
add/
 ├─ registry.json              : 확장 기능 목록 DB
 ├─ installed/                 : 실제 확장 기능 저장
 ├─ config/                    : 전역 설정 (환경 변수 참조)
 ├─ runtime/                   : 실행 중 생성 임시 데이터
 └─ template/                  : 신규 확장 기능 템플릿
report/
 ├─ fragment/                  : Sub AI 개별 작업 보고서 저장
 └─ report.md                  : Collector AI가 취합한 최종 보고서

9. 확장 기능 관리 흐름 (Extension Management Flow)
==================================================

[/installExtension [url]]

Boss AI: git clone [url] → add/installed/[name]/
  → extension.json 읽기
      (포함 항목: name, version, commands, mainAI, subAI, requires)
  → add/registry.json에 등록 (status: "active")
  → extension.json에 mainAI 정의된 경우:
       HR AI → AI_list.txt에 해당 Main AI 항목 추가
  → 유저에게 설치 완료 및 신규 명령어 목록 보고

[오류 처리]
  - URL 접근 불가         → clone 실패 보고, 설치 중단
  - 동일 name 이미 존재   → 덮어쓰기 여부 유저에게 확인
  - extension.json 없음   → 설치 중단, 올바른 확장 기능 레포가 아님을 보고

--------------------------------------------------

[/removeExtension [name]]

Boss AI: add/registry.json에서 [name] 확인
  → AI_list.txt에서 해당 확장의 Main AI가 STATUS: WORKING인지 확인
       WORKING 존재 시 → 유저에게 경고 후 확인 받기
  → add/installed/[name]/ 폴더 삭제
  → add/registry.json에서 항목 제거
  → extension.json에 mainAI 정의된 경우:
       HR AI → AI_list.txt에서 해당 Main AI 항목 제거
  → 유저에게 제거 완료 및 비활성화된 명령어 목록 보고

[오류 처리]
  - [name]이 registry에 없음 → 설치된 확장 기능이 아님을 보고

--------------------------------------------------

[/listExtensions]

Boss AI: add/registry.json 읽기
  → 설치된 확장 기능 없음 → "설치된 확장 기능이 없습니다" 출력
  → 있는 경우 아래 형식으로 출력:

  [name] v[version] — [status]
    명령어: /cmd1, /cmd2, ...
    Main AI: [이름] (없으면 생략)

--------------------------------------------------

[extension.json 명세]

확장 기능 레포 루트에 반드시 포함해야 하는 파일.

{
  "name"     : "확장 기능 식별자 (영문 소문자, 하이픈 허용)",
  "version"  : "버전 문자열 (예: 1.0.0)",
  "commands" : ["/명령어1", "/명령어2"],
  "mainAI"   : "추가될 Main AI 이름 (없으면 null)",
  "subAI"    : ["추가될 Sub AI 역할 목록"],
  "requires" : {
    "server" : true | false,
    "db"     : "postgresql" | "sqlite" | null
  },
  "specPath" : "AI 역할 명세 파일 경로 (예: ./info.txt)"
}
